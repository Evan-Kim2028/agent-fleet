## Dependency-driven scheduling proof (Scenario 6)

The DAG has two independent root tasks: `root-F` (fast, 0.02s) and `root-S` (slow, 0.25s).
`leaf-F` depends only on `root-F` (via `mid-F`). A rank-barrier scheduler would make
`leaf-F` wait for the entire rank-0 to finish, i.e., wait for `root-S`.

Observed timestamps (one representative run):

```
root-S  start=T+0.000  end=T+0.250
leaf-F  start=T+0.041  end=T+0.062   ← finishes 0.189s before root-S
```

`leaf-F.end < root-S.end` — confirmed. The scheduler launches each task the moment its own
dependencies are satisfied, not when the whole topological rank drains.

## v0.8.6 vs v0.9.0 tokens-to-parent

**Workload:** a parent task whose work decomposes into N=20 child sub-tasks, each producing a
~2000-char result, whose outputs must be combined into a single answer.

**Measurement method:** the v0.8.6 number is a real measurement. Twenty `FleetTaskResult`
objects (each with a 2000-char summary) are constructed and passed to `aggregate_child_results()`
(`agent_fleet/orchestration/decompose.py:199-226`). That function is the actual aggregation seam:
it builds the summary string the parent receives, including a header line per child
(`decompose.py:213`) and up to 200 chars of each child's summary (`decompose.py:215`). The length
of the returned string is divided by 4 to estimate tokens — the same formula used throughout the
codebase. The v0.9.0 number comes directly from `ProgramRunSummary.tokens_to_parent`, which
`run_workflow_program()` computes as `len(stringify(result)) // 4` — only the synthesised return
value crosses back to the parent.

| Path | tokens_to_parent | How measured |
|------|----------------:|--------------|
| v0.8.6 decompose | **1 353** | `aggregate_child_results()` on 20×FleetTaskResult(summary=2000 chars); summary string = 5 415 chars → 5415//4 = 1 353 |
| v0.9.0 program   | **14**    | `ProgramRunSummary.tokens_to_parent`; synthesised result = 58 chars → 58//4 = 14 |
| **Reduction factor** | **96.6×** | 1 353 / 14 |

**Why v0.8.6 is so expensive:** `aggregate_child_results` appends up to 200 chars of each
child's summary plus a status header line for every child, so parent cost grows linearly with
N. At N=20 the parent must read the full aggregated block — a 5 415-char wall.

**Why v0.9.0 is bounded:** the program returns a single synthesised string (~58 chars here).
Each subagent's full transcript stays inside its own context window; only the program's `return`
value crosses back. The parent context cost is O(1) in the number of agents.

Test: `tests/stretch/test_compare_v086_v090.py` — asserts `reduction_factor > 10` (observed 96.6×).
