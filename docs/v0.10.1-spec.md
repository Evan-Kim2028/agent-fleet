# v0.10.1 — observability polish

A small follow-up to v0.10.0 (`5cde796 chore(release): bump v0.9.0 -> v0.10.0
(capacity-aware fanout)`). v0.10.0 shipped the dispatch consolidation and
capacity-aware fanout; v0.10.1 closes three small parity gaps in the
observability layer and documents the two-stream JSONL split so future analyzers
don't mistake one for the other.

## Scope

This release is observability-only. No new features. No critical-path behavior
change. The dispatcher's task-execution flow, the runner's pipeline, and the
cursor backend are all untouched at the call-site / control-flow level.

## What changed

### 1. `build_run_metrics()` parity across all four production call sites

`agent_fleet/observability/run_metrics.py::build_run_metrics()` accepts ten
optional kwargs. Before v0.10.1, only two of the four call sites
(`runner.py:150` and `level_up/record.py:192`) passed the full set. The three
remaining sites were each missing one or two kwargs, so the metric dict's
`repo_key`, `issue_number`, `duration_seconds`, and `error` fields could be
silently absent depending on which path emitted them.

Closed in v0.10.1:

- `agent_fleet/dispatcher.py:472` — added `issue_number=task_index` to the
  `build_run_metrics(...)` call inside `_execute_task`.
- `agent_fleet/dispatcher.py:730` — added `repo_key=level_up_repo_key(...)`
  pass-through to `build_task_result(...)` (and a corresponding `repo_key`
  kwarg + inner-call propagation in `dispatcher_task.py:235,285`).
- `agent_fleet/dispatcher.py:768,797` — both `record_completed_task_experience`
  call sites now pass `duration_seconds` and `error`, which propagate through
  `record_completed_task_experience` (`dispatcher_task.py:308`) into
  `record_task_experience` (`level_up/record.py:88,117`) and on into the
  `build_run_metrics(...)` call there.

All changes are additive kwargs on existing call sites. No public API change.
No behavior change for paths that already passed full kwargs.

**Why this is safe even though `dispatcher.py` is in `critical_path_prefixes`.**
The `critical_path_prefixes` allowlist in `.agent-fleet.yaml` gates *fleet-
dispatched persona edits* — it prevents an autonomous coder run from rewriting
the dispatcher. It is not a hard prohibition on human dev edits. v0.10.1 was
authored locally by the maintainer, kept to additive kwargs, runs the full
suite green (`uv run pytest -q`: 575 passed, 4 skipped), and passes
`uv run ruff check .` clean.

### 2. `docs/OBSERVABILITY.md` — two-stream JSONL clarification

The runs directory contains two co-located JSONL streams with different
schemas:

- `<run_id>.jsonl` — the strict FleetEvent stream. `event` is always a string,
  one of a fixed dotted-namespace event name set. Producer: `RunLog.emit()`.
- `<run_id>.bridge.jsonl` — raw SDK passthrough. `event` is a **dict** built by
  reflecting public attributes off the Claude Code bridge's update object.
  Producer: `cursor_backend._write_bridge_event()`.

This was previously implicit; a casual reader scanning the runs dir could
reasonably think the dict-shaped `event` records were a bug. They are not — the
two streams serve different consumers and the separation is intentional.
`docs/OBSERVABILITY.md` documents the schema, the canonical event name set, and
the reasoning for keeping the bridge stream out of the main file.

### 3. `tests/test_event_schema_guard.py`

Four-case guard for the FleetEvent main-stream contract:

1. `FleetEvent.event` is typed `str` (via `typing.get_type_hints`).
2. `RunLog.emit()` produces JSONL lines whose `event` parses back as a string.
3. `JsonlFileSink` does not leak extra keys beyond `FleetEvent.to_dict()`.
4. Live sweep — when `default_runs_dir()` exists, walks the 5 most recent
   `*.jsonl` files (excluding `*.bridge.jsonl`) and confirms every record's
   `event` is a string. Skipped when the runs dir is absent or empty.

The live sweep is what catches a future mistake where someone routes raw
bridge records into the main file or widens the `FleetEvent.event` type to
`str | dict`.

## What this release explicitly does NOT do

- **Bridge instability** — diagnosed in the v0.10.1 audit but
  critical-path-blocked. Needs a separate release with its own design doc.
- **New event types** — no additions to the canonical event name set. The doc
  reflects the current set as of v0.10.1.
- **Schema migration** — no field renames, no removals. The guard test pins
  what exists today.
