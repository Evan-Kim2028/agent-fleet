## Candidate 5 — public dispatch-primitives adapter

**Problem.** The single-task primitive `FleetDispatcher._execute_task` is reached
through two near-identical `ThreadPoolExecutor` + wave-batching + admission blocks,
one in `orchestration/decompose.py` and one in `orchestration/dag/runner.py`. The
batching wiring is duplicated; only the per-task call is shared.

**Solution.** Add `agent_fleet/orchestration/primitives.py` with one
admission-controlled batch dispatcher:

```python
class DispatchPrimitives:
    def __init__(self, dispatcher: _DispatcherLike, *, max_parallel: int) -> None: ...
    def run_one(self, index: int, task: FleetTask) -> FleetTaskResult: ...
    def run_many(self, tasks: list[FleetTask]) -> list[FleetTaskResult]: ...
```

- `_DispatcherLike` is a `Protocol` with the existing `_execute_task` signature
  (reuse the one already defined in `orchestration/program/runtime.py`; lift it to
  a shared spot under `orchestration/` if cleaner). The adapter NEVER edits
  `dispatcher.py`; it accepts any object exposing `_execute_task`.
- `run_many` owns the ThreadPoolExecutor + wave-batching + admission logic that is
  currently copy-pasted. Preserve same-workspace batching and ordering of results
  by task index.
- Migrate `decompose.py` and `dag/runner.py` to build a `DispatchPrimitives` and
  call `run_many` instead of their own executor blocks. Behavior must be identical
  (the existing suite is the regression gate).

## Candidate 3 — unified output convergence

**Problem.** "Combine N subagent results" exists in four shallow forms with
copy-pasted success vocabulary: `decompose.aggregate_child_results`,
`dag.runner.aggregate_dag_results` (two status-rollup functions with duplicated
`_SUCCESS_STATUSES` / `_PARTIAL_OK` / `_FAILURE_STATUSES` sets),
`synthesizer.synthesize` (LLM collapse), and `dag.stitch.build_dag_task_context`
(truncate-to-2000). The vocabulary leaks across the codebase.

**Solution.** Add `agent_fleet/orchestration/convergence.py` as the single source of
truth:

- One canonical set of status constants (`SUCCESS_STATUSES`, `PARTIAL_OK`,
  `FAILURE_STATUSES`) and one `roll_up_status(results) -> str` function.
- One `aggregate(results, *, max_chars_per_child) -> tuple[str, str | None, str]`
  that builds the parent-facing summary string (the shape both aggregation
  functions already return).
- Migrate `decompose.aggregate_child_results` and `dag.runner.aggregate_dag_results`
  to delegate to `convergence`. Keep their public signatures unchanged.
- Migrate `dag/stitch.build_dag_task_context` to import the shared status constants
  rather than redefining them.
- Leave `synthesizer.synthesize` (top-level, outside orchestration/) untouched in
  this pass; note it in the PR body as a follow-up. Do not edit `synthesizer.py`.
