# v0.10.0 — optimized convergence + dispatch consolidation

v0.9.0 shipped the PROGRAM paradigm, the WorkflowProgram runtime, and the
PersonaFoundry. v0.10.0 finishes the architecture-review backlog (candidate 5 =
public dispatch primitive, candidate 3 = unified output convergence) AND folds in
the highest-value token-efficiency optimization a perf analysis confirmed. The
result is a fleet that combines N subagent results into a parent-facing summary
whose size is bounded and independent of N. Everything lands inside
`agent_fleet/orchestration/` so no bootstrap-trap file is touched.

This build runs against the v0.9.0 base (the worktree is cut from `main`, now at
the v0.9.0 commit). The PROGRAM runtime, PersonaFoundry, and the stretch suite
under `tests/stretch/` are present and must stay green.

## Candidate 5 — public dispatch-primitives adapter

**Problem.** The single-task primitive `FleetDispatcher._execute_task` is reached
through two near-identical `ThreadPoolExecutor` + wave-batching + admission blocks,
one in `orchestration/decompose.py` and one in `orchestration/dag/runner.py`. Only
the per-task call is shared; the batching wiring is copy-pasted.

**Solution.** Add `agent_fleet/orchestration/primitives.py`:

```python
class DispatchPrimitives:
    def __init__(self, dispatcher: _DispatcherLike, *, max_parallel: int) -> None: ...
    def run_one(self, index: int, task: FleetTask) -> FleetTaskResult: ...
    def run_many(self, tasks: list[FleetTask]) -> list[FleetTaskResult]: ...
```

- `_DispatcherLike` is a `Protocol` exposing the existing `_execute_task`
  signature. Lift it into `agent_fleet/orchestration/types.py` as the single
  shared definition and have `program/runtime.py` import it from there. The
  adapter NEVER edits `dispatcher.py`; it accepts any object exposing
  `_execute_task`.
- `run_many` owns the ThreadPoolExecutor + wave-batching + admission logic that is
  currently duplicated. Preserve same-workspace batching and ordering of results
  by task index.
- Migrate `decompose.py` and `dag/runner.py` to build a `DispatchPrimitives` and
  call `run_many` instead of their own executor blocks. Behavior must be
  identical. The existing suite is the regression gate.

## Candidate 3 — unified output convergence with bounded summaries

**Problem.** "Combine N subagent results" exists in four shallow forms with
copy-pasted success vocabulary: `decompose.aggregate_child_results`,
`dag.runner.aggregate_dag_results`, `synthesizer.synthesize` (leave untouched),
and `dag.stitch.build_dag_task_context`. Worse, the aggregate summary is O(N): it
appends a header line plus up to ~200 chars per child. At N=20 the parent-facing
summary is ~6000 chars, and in nested DAG topologies that bloat re-enters the
token budget at every level.

**Solution.** Add `agent_fleet/orchestration/convergence.py` as the single source
of truth, with a bounded summary:

- One canonical set of status constants (`SUCCESS_STATUSES`, `PARTIAL_OK`,
  `FAILURE_STATUSES`) and one `roll_up_status(results) -> str`.
- `compact_summary(results, *, total_chars=400) -> str`. The parent-facing
  summary is BOUNDED, not O(N). All-success collapses to a count line (for
  example `"20/20 completed"`). Partial or failing runs add up to 3 failed
  children, each `goal[:60]` plus `error[:100]`. The whole string stays under
  `total_chars`.
- `aggregate(results, *, max_chars_per_child) -> tuple[str, str | None, str]`
  keeps the public shape the two functions already return (status, error-or-None,
  summary) but builds the summary via `compact_summary`.
- `budget_upstream_context(outputs, dep_ids, *, total_budget=2000) -> str` for the
  DAG stitch path. The TOTAL upstream context is capped at `total_budget`, divided
  across dependencies, instead of a fixed budget PER dependency. A single-dep node
  keeps its full 2000-char budget; a 10-dep node no longer balloons to 20000.
- Migrate `decompose.aggregate_child_results` and `dag.runner.aggregate_dag_results`
  to delegate to convergence. Keep their public signatures unchanged.
- Migrate `dag/stitch.build_dag_task_context` / `build_upstream_context` to import
  the shared status constants and use `budget_upstream_context`.
- Leave `synthesizer.synthesize` (top-level, outside orchestration/) untouched.
  Note it in the PR body as a follow-up.
