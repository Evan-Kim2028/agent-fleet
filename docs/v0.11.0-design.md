# agent-fleet v0.11.0 design — wider dynamic surface, elastic bound, durable event log

Synthesized design package from the architect/arena pass. Three isolated candidates
(`/tmp/v0.11.0-arena/candidate-{1,2,3}.md`, summarized in the decision trail) explored
whole-shape alternatives across three contested axes. This is the chosen base plus grafts.

## Problem

agent-fleet v0.10.0 is a single-process Python 3.14 fleet on the Cursor SDK. Four weak
points cap its autonomy. (1) A crash loses an in-flight run; the JSONL event stream is an
audit trail, not a recoverable state log. (2) The capacity bound is hand-copied as
`effective_capacity(reserved=1)` in three producers (`program/runtime.py`,
`dag/runner.py`, `decompose.py`); the invariant can drift. (3) The dynamic control surface
is narrow. The program runtime exposes five primitives (`agent/parallel/pipeline/phase/log`)
and `dag` is static and acyclic; neither expresses a model-triggered re-plan loop, a
model-judged conditional branch with a guaranteed-legal verdict, or bounded sub-program
recursion. (4) Observability is human-readable logs, not a queryable stream.

Constraints the design must honor, taken from the grounding. The 4GB/agent RAM admission
ceiling stays exact; overflow queues, never denies. The `AdmissionController` stays the
single owner of RAM accounting. `TaskSpec` keeps its `dag | program` seam. The program
runtime keeps its restricted-namespace AST-validated execution model. The change lands as
independent units, critical-path files edited by the integrator, the rest dogfooded.

## Shape

Three separable concerns, each grafted from the candidate that solved it best, none
coupled to the others more than the data forces.

### Elastic bound — `CapacityGate` wraps the real controller (Unit 3)

Data first. `CapacityGate` holds the existing `AdmissionController` and a
`threading.Condition`. `acquire()` is a context manager that blocks until
`AdmissionController.try_admit("agent")` returns a token, yields it, and on exit releases
the token and notifies waiters. One gate is constructed on the dispatcher; all three
producers call it instead of computing a pool ceiling.

The load-bearing decision: because the gate wraps the *same* controller the parent was
admitted through, the parent's 4GB is already counted in `in_flight_ram_gb`. A child that
would exceed the budget blocks on the condition until the parent or a sibling releases.
The `reserved=1` arithmetic does not move to one place. It stops existing, because the
shared RAM accounting already reserves the parent's slot (per redesign-from-first-principles
and single-source-of-truth). Overflow queues on the condition variable rather than being
denied (the missing-correct-behavior the v0.10.0 `reserved=1` patch worked around).

Rejected: Candidate 2's asyncio `CentralScheduler` (a queue+event-loop rewrite is too much
blast radius for a sync codebase) and Candidate 3's `BoundedWorkPool` (it reimplements the
`min(max_parallel, ram//4)` math with its own counter, dropping the controller as the RAM
source of truth). The gate keeps the controller authoritative and stays synchronous.

### Durable event log — typed append-only JSONL, state is a fold (Units 2 and 5)

Data first. A `RunEvent` is a frozen record with a `kind` discriminant from a sealed set, a
`run_id`, a monotonic `seq`, a deterministic `key`, and a `data` payload. Events append to a
dedicated per-run JSONL journal, emitted through the existing `FleetLogger` seam so we reuse
the sink rather than build a broker. Run state is a pure `fold(events) -> RunState`. Crash-
resume replays the fold, re-drives agents that were dispatched but never finished, and serves
already-finished agents from their journaled result instead of re-dispatching. Observability
is a query over the same fold (by run, by status, in-flight set, timeline).

The load-bearing idempotency decision, grafted from Candidates 2 and 3: every nondeterministic
control decision (a `branch` verdict, a `replan` output) is journaled, and on resume the
recorded decision is replayed rather than recomputed. That makes a program with model-judged
control flow deterministic on replay, so the fold's "which agents already finished" is exact.
Agent identity is the stable key `(run_id, dispatch_index)`, deterministic for a given program.

Rejected: Candidate 3's SQLite store (a new dependency, NFS-unsafe by its own open question,
per-event commit cost) when a typed JSONL journal folded in memory is queryable enough for the
predicate. A SQLite projection can be a derived index later, never the source. Also rejected:
Candidate 1's piggyback of journal events into the shared human-readable log file (couples the
durability contract to the observability file). The journal is its own file; it reuses the
emit seam, not the file.

### Wider control surface — three bounded primitives, closed-key branch (Unit 4)

Data first. Three callables join the program namespace, each bounded by a hard cap tracked on
`ProgramContext` and recorded in the journal at run start so the bound is durable data, not
convention.

- `replan(results, *, hint, max_iterations)` re-invokes the planner agent with accumulated
  results to get a new validated sub-program, then runs it nested. The iteration count is
  journaled; exceeding `max_iterations` raises.
- `branch(judge_prompt, branches, *, default)` where `branches: dict[str, Callable]`. A judge
  agent runs with a schema constraining its verdict to `branches.keys()`; the matching thunk
  runs; an unrecognized verdict falls to `default`. The closed-key set is grafted from
  Candidate 3's `PredicateNode.branches`: it makes a hallucinated verdict a defined fallback,
  not a silent wrong path (per encode-invariants-in-types). The verdict is journaled.
- `subprogram(source, *, depth=auto)` validates source and runs it nested with
  `_subprogram_depth + 1` checked against `MAX_SUBPROGRAM_DEPTH`. Bounded recursion.

The decision: extend the program runtime that already *is* the dynamic seam rather than add a
third execution path. The program string already expresses arbitrary Python control flow; we
expose validated, bounded, journaled higher-order forms of it. This delivers the full predicate
(loop, conditional branch, bounded recursion) at a fraction of Candidate 3's blast radius (per
laziness-protocol and subtract-before-you-add).

Rejected for v0.11.0: Candidate 3's first-class typed `GraphSpec`. It is more expressive and
statically analyzable, and it is the right evolution *if* the string-plus-primitives approach
proves unable to hold the invariants. That is the explicit architect-Phase-E scrap signal: a
repeated pattern of control-flow correctness friction the primitives cannot absorb. We take the
smaller bet first and keep the graph as the named fallback.

### Per-task skill selection (Unit 1)

The `Persona`/`PersonaSpec`/`personas.load` skill-slot machinery already exists
(`skill_slots_execute`, `skill_slots_review`, `merge_skill_dirs`). Unit 1 vendors
`addyosmani/agent-skills` into the base kit and lets the planner select skills per task into
the existing slots. No new substrate. Independent of the core, dogfoodable, can land early.

## Synthesis decision

Base: Candidate 1 (minimal seam), for its unit decomposition and smallest blast radius.

Grafted from Candidate 3: the closed branch-key set with a default verdict (kills the
unguarded-branch failure mode Candidate 1 carried), and the typed sealed-event discipline for
the journal.

Grafted from Candidate 2: run state as a pure fold over an append-only journal, deterministic
event keys, caps recorded as durable data at run start, and the journal-verdict-replay
idempotency mechanism that makes resume exact.

Rejected: Candidate 2's asyncio `CentralScheduler` (blast radius), Candidate 3's full graph
subsystem and SQLite store (more than the predicate needs; deferred as the Phase-E fallback),
Candidate 1's at-least-once re-dispatch and shared-file journal (weak idempotency; replaced by
fold + journaled-verdict replay in a dedicated file).

## Tradeoffs accepted

- We accept synchronous `CapacityGate.acquire()` parking a thread on a condition variable in
  exchange for keeping the controller authoritative and avoiding an asyncio rewrite. A wide
  fan-out parks N threads; N is tens of agents here, which is fine.
- We accept a dedicated second JSONL file per run (the journal) in exchange for not coupling
  durability to the human-readable log.
- We accept that `replan` and the `branch` judge each spend one agent slot through the shared
  gate in exchange for no separate meta-agent lane. Flagged as an open question.
- We accept program-string control flow over a statically analyzable graph for v0.11.0 in
  exchange for a far smaller change that still meets the predicate, with the graph named as the
  Phase-E fallback.
- We accept agent-level resume granularity (re-drive in-flight agents, replay journaled
  verdicts) rather than control-flow-node granularity, in exchange for a fold that stays a small
  pure function.

## Alternatives considered

- First-class typed execution graph (Candidate 3). More expressive and statically checkable;
  lost on blast radius for v0.11.0 (a third execution path plus a new `TaskSpec.graph` field
  plus SQLite) when three bounded primitives meet the predicate. Retained as the Phase-E
  scrap-and-regrow target.
- Central asyncio scheduler owning one queue (Candidate 2). Cleanest single-owner story for the
  bound; lost because porting a synchronous dispatcher and three producers to asyncio is a
  larger and riskier change than wrapping the existing controller in a blocking gate.

## Open questions and risks

- Should `replan`/`branch` judge agents reserve a slot outside the gate so meta-decisions never
  starve behind child work, or is sharing the gate acceptable at this scale?
- Is `(run_id, dispatch_index)` a stable enough agent key across a resumed run where `replan`
  changed the program? The journaled-verdict replay is meant to hold dispatch order fixed; this
  is the central resume risk and the reason resume (Unit 5) lands after the log and primitives
  are observed working.
- Are any fleet task types non-idempotent at the agent level (PR creation, issue comments) such
  that even a correctly-resumed re-drive double-posts? If so they need a journaled completion
  guard.
- The AST validator (`program/validate.py`) must allow the three new names; what is the minimal
  allowlist change, and do the primitives need recursion-depth guards in the validator too?

## Unit plan (figure-it-out Phase B output, riskiest-unknown-first)

- Unit 0 — verification harness + baseline. Measures control-surface width, denials-under-load,
  and absence of crash-resume from the pre-change state, so each later check reads old-vs-new.
  Integrator. Lands first.
- Unit 3 — `CapacityGate`. Centralize the elastic bound, blocking acquire wrapping the real
  controller, delete the three hand-copied `reserved=1` sites. Riskiest core change; unblocks
  confident load testing. Integrator (critical-path).
- Unit 2 — typed append-only JSONL run journal + fold/query. Emission touches the dispatcher
  (integrator); the fold and query reader are non-critical-path (dogfood).
- Unit 1 — vendor `addyosmani/agent-skills` + per-task skill selection via existing slots.
  Low-risk, independent, dogfoodable; can run in parallel with the core via the fleet.
- Unit 4 — widen the control surface: `replan`, `branch` (closed-key), `subprogram`, plus the
  validator allowlist. Integrator (critical-path). The headline autonomy win.
- Unit 5 — crash-resume: fold the journal, replay journaled verdicts, re-drive in-flight,
  memoize finished agents. Integrator (critical-path). Depends on Units 2 and 4.
- Unit 6 (optional) — adversarial self-review gate wiring `doubt-driven-development` +
  `code-review-and-quality`. Dogfoodable.

## Done predicate (falsifiable, verified on the real fleet)

1. The orchestrator expresses a re-plan loop, a model-judged conditional branch, and per-task
   skill selection, each shown end-to-end on the real product.
2. A load test at the widened surface shows zero admission denials; overflow queues.
3. A crash-resume test recovers an in-flight run from the journal to the same end state.
4. The event log is queryable (by run, status, in-flight set, timeline).

## Next implementation step

Build Unit 0: the verification harness that captures the pre-change baseline for all four
predicate dimensions, so the `CapacityGate` and control-surface work that follows reads as a
measured old-value-versus-new-value delta.
