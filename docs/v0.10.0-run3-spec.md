# v0.10.0 run 3 — capacity-aware fanout (parent-slot aware)

Run 2 delivered bounded convergence and the dispatch-primitives adapter (the
token-efficiency half of the optimization). This run finishes the
parallelization-and-fanout half.

## The bug (reproduced, not theoretical)

The dispatcher admits agents through a NON-blocking gate:
`FleetDispatcher._execute_task` calls `self._admission.try_admit("agent")`, and
when the `agent` tier is full it returns immediately with
`status="error"`, `error="Fleet admission denied (max parallel agents reached)"`.
It does not wait or queue. Tier capacity is
`min(config.max_parallel, config.ram_budget_gb // 4)` (the agent tier reserves
4 GB each). Defaults are `max_parallel=3`, `ram_budget_gb=24`, so capacity is 3.

**Orchestration always runs nested inside an already-admitted parent.** This is
the crux. `_execute_task` admits the parent token (dispatcher.py:558), then calls
`_maybe_preflight_and_dispatch` (dispatcher.py:643) which runs `dispatch_dag` /
`dispatch_task_spec_children` / `run_workflow_program`, and only releases the
token afterward (dispatcher.py:818). So while a producer fans out children, the
parent already holds ONE of the `agent` slots. The children can therefore admit
only `capacity - 1` at a time, not `capacity`.

Two producers submit MORE concurrent dispatches than the gate admits, so overflow
fails with spurious admission-denied errors instead of running:

1. `orchestration/dag/runner.py::dispatch_dag` builds its
   `ThreadPoolExecutor(max_workers=max(1, n_total))` and submits every ready
   task. A rank wider than capacity oversubscribes.
2. `orchestration/program/runtime.py` `parallel()` / `pipeline()` clamp workers
   to `self._max_parallel`, whose default is 4 (the `run_workflow_program`
   default; the production caller in `dispatcher.py` passes no override). 4 > the
   default capacity, so a wide `parallel()` denies its overflow.

Because admission is non-blocking, an over-provisioned worker does not wait. It
gets denied instantly and the pool immediately feeds it the next ready task,
which is also denied. One surplus worker becomes a denial mill. Measured baseline
under the real nested condition (parent holds 1 token, width 8, capacity 3): 2
completed, 6 admission-denied, status `dag_partial`. Bounding the pool to
`effective_capacity` WITHOUT reserving the parent slot does not fix this: it
still launches `capacity` workers against `capacity - 1` free slots, so the
surplus worker still churns to 6 denied. The fix must reserve the parent slot.

`decompose.py` already bounds its waves to `dispatcher.config.max_parallel`, but
it has the same parent-slot blind spot (it is also reached from
`_maybe_preflight_and_dispatch`), so it gets the same reservation.

`same_workspace_tasks` stays 1 on the DAG path. That is intentional and correct
(DAG nodes compose into the single parent workspace; `worktree.py` only forces
parallel-safe isolation when `batch_size > 1 AND same_workspace_tasks > 1`). Do
not change it.

## The fix — one concept, applied to every producer

Producers bound their concurrency to the dispatcher's effective admission
capacity MINUS the slot the parent already holds, so overflow QUEUES in the
thread-pool work queue instead of hitting the non-blocking gate and failing. No
deadlock is possible: the DAG is acyclic and a child is released only after its
parents complete, so no running task ever waits on a queued sibling.

### 1. Single source of truth — `effective_capacity`

Add to `agent_fleet/orchestration/primitives.py`:

```python
_RAM_GB_PER_AGENT = 4  # mirrors admission.ResourceTier("agent", ram_gb=4)

def effective_capacity(dispatcher: object, *, fallback: int, reserved: int = 0) -> int:
    """Max NEW agents the dispatcher's admission gate will admit at once.

    Reads dispatcher.config when present (the real FleetDispatcher); uses
    `fallback` for stand-ins (test doubles) that expose no config. `reserved`
    is the number of agent slots already held by callers up the stack (the
    orchestrating parent holds 1), so children never oversubscribe the gate.
    """
```

- Use `getattr(dispatcher, "config", None)` — do NOT add `config` to the
  `_DispatcherLike` Protocol (test doubles have no `config` and must stay
  conformant; getattr keeps the type checker quiet).
- Compute the base capacity: when a config is present, `cap = config.max_parallel`
  and if `config.ram_budget_gb` is set, `cap = min(cap, config.ram_budget_gb //
  _RAM_GB_PER_AGENT)`; when no config, `cap = fallback`.
- Return `max(1, cap - reserved)`. The `max(1, ...)` floor guarantees forward
  progress even at `capacity == 1` (degenerate serial config: at most one child
  at a time, which is the most the gate could ever allow anyway).

### 2. DAG runner — bound the pool, reserve the parent

In `orchestration/dag/runner.py::dispatch_dag`:

- Compute `inflight = max(1, min(n_total, effective_capacity(dispatcher,
  fallback=n_total, reserved=1)))`. The `reserved=1` is the parent token held by
  the `_execute_task` that invoked this DAG.
- Use `inflight` for the `ThreadPoolExecutor(max_workers=...)`. (The
  `DispatchPrimitives(max_parallel=...)` there is only used via `run_one`, which
  ignores `max_parallel`; set it to `inflight` too for clarity but the pool is
  what matters.)
- Update the comment to state: concurrency is bounded to admission capacity less
  the parent's held slot; the rest of the rank queues.

### 3. PROGRAM runtime — clamp + raise the ceiling

In `orchestration/program/runtime.py`:

- Cache capacity once: in `__init__`, `self._capacity =
  effective_capacity(dispatcher, fallback=self._max_parallel, reserved=1)`. The
  parent `_execute_task` that launched the program holds one slot.
- In `parallel()` and `pipeline()`, clamp workers with the capacity too:
  `workers = max(1, min(self._max_parallel, self._capacity, len(items)))`.
- Raise the `run_workflow_program` `max_parallel` default from 4 to 16. This is
  now SAFE because the capacity clamp prevents oversubscription: at default
  config the effective worker count is still capacity-bound, but when an operator
  raises `config.max_parallel` (and RAM) the runtime fans out wider instead of
  being pinned at 4.

### 4. decompose — route through the helper, preserve explicit overrides

In `orchestration/decompose.py`, where it currently does
`limit = max_parallel or dispatcher.config.max_parallel`, compute
`limit_raw = max_parallel or dispatcher.config.max_parallel` and then
`limit = min(limit_raw, effective_capacity(dispatcher, fallback=limit_raw, reserved=1))`.
The `min` preserves an explicit smaller `max_parallel` override (a caller asking
for narrower waves), while the helper caps it at the parent-aware admission
capacity. Behavior at default config drops from 3 to 2, which is correct: the
decompose parent holds the third slot.

## Tests — add `tests/test_capacity_fanout.py`

Use a stand-in dispatcher whose `_execute_task` honors a REAL
`admission.AdmissionController` (so the gate behaves exactly like production:
non-blocking, returns `status="error"` "admission denied" when full). Track peak
concurrent admitted count under a lock.

**The stand-in test MUST model the nested production path: acquire one `agent`
token from the dispatcher's own `AdmissionController` BEFORE calling
`dispatch_dag` / running the program, hold it across the dispatch, and release it
after.** A test that dispatches with no parent token gives a FALSE PASS — it
leaves the full `capacity` free and never exercises the reservation. This is the
exact gap that hid the bug; the test is the fix's proof, so it must hold the
parent slot.

1. `effective_capacity`: config `(max_parallel=3, ram=24)`, `reserved=0` -> 3;
   `reserved=1` -> 2; config `(max_parallel=10, ram=24)`, `reserved=0` -> 6
   (RAM-bound); object with no config -> `max(1, fallback - reserved)`; the
   `max(1, ...)` floor holds when `reserved >= cap`.
2. **Drift guard** (encode the lesson so the duplicated constant cannot rot):
   build a real `FleetDispatcher()` and assert
   `primitives._RAM_GB_PER_AGENT` equals its agent tier's `ram_gb`
   (`dispatcher._admission._tiers["agent"].ram_gb`).
3. **Nested DAG wide rank**: hold one parent `agent` token, then 8 independent
   tasks at capacity 3. Assert all 8 `completed`, ZERO admission-denied,
   `aggregate_status == "completed"`, and peak concurrent children `<= 2`
   (capacity minus the parent slot). Baseline before this change: 6 denied,
   `dag_partial`.
4. **Negative control** (prove the fix is load-bearing): the same nested wide
   rank with the pool deliberately unbounded (`max_workers = n_total`, or
   `effective_capacity` monkeypatched to return `n_total`) MUST still reproduce
   admission-denied failures. If the negative control passes, the test is not
   actually exercising the gate.
5. **Nested runtime fanout**: hold one parent token, run a program (or direct
   `parallel()` of dispatching thunks) of 8 tasks. At capacity 3: zero
   admission-denied. At capacity 6: peak concurrent children reaches `> 3`
   (fanout scales with capacity once the parent slot is accounted for).

For every concurrency assertion, assert on COUNTS / SETS / PEAK, never on
intra-wave execution ORDER. Parallel waves finish in nondeterministic order;
order assertions are flaky (a run-2 defect we will not repeat). Run any new
concurrency scenario ~30x to confirm 0 flakes — prefer an internal loop inside
one test over 30 parametrized cases, to keep the suite lean.
